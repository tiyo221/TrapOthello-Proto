# CODING.md — コーディング規約

このリポジトリのコードを書くときの規約（**どう作るか**）。**何を作るか**は [`spec.md`](spec.md)、開発ワークフローは [`CLAUDE.md`](CLAUDE.md)。

設計思想（重要）: **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない。ファイルは責務で分割するが、`file://` で確実に動かすため **ES Modules / `fetch()` は使わず**、classic `<script src>` ＋ グローバル名前空間 `window.TO` で連携する（構成は [`spec.md`](spec.md) §4）。画像はインライン SVG か CSS、音・フォント・ライブラリを外部から読み込まない。

## 言語・実行環境
- Vanilla JS（ES2020+）。フレームワーク・トランスパイラ・バンドラを使わない。実行時に npm パッケージを含めない。
- ES Modules（`import` / `type="module"`）と `fetch()` は使わない（`file://` のため）。公開シンボルは `window.TO` 名前空間へ集約する。
- 各ファイルは IIFE ＋ `"use strict"` で包み、必要なものだけを `TO.<name>` に代入して公開する。ファイル内のヘルパをグローバルへ漏らさない。

## モダン JS
- **`var` を使わない。** `const` を既定とし、再代入が必要なときだけ `let`。
- 比較は `===` / `!==`（`==` / `!=` を使わない）。
- イベントは `addEventListener` で登録する（HTML 属性の `onclick=` 直書き・`document.write` を使わない）。
- 反復は `for...of` / 配列メソッドを基本とする（盤面の 64 マス走査などインデックス操作が自然な箇所は無理に置き換えない）。

## 構造（レイヤーの契約）

[`spec.md`](spec.md) §4.1 の依存方向を守る。要点:

- **状態は `game.js` が一元的に持つ**。他のレイヤーは対局状態を複製せず `TO.game.state()` を読む。
- **`rules.js` / `game.js` / `cpu.js` は DOM に触れない。** `document` / `window` の API を参照しない（そのまま Node で動かせる純粋さを保つ）。
- **`view.js` は判定をしない。** 勝敗・合法手・罠の解決は `rules` / `game` に委ね、描くだけにする。
- **演出のタイミング（`setTimeout`）は `main.js` に集約する。** `view.js` / `game.js` に進行制御を持ち込まない（例外は CSS アニメーションの後始末）。
- **調整値は `config.js` に集約する。** マジックナンバー（枚数・手数・待ち時間・評価の重み）をロジック側に直書きしない。
- 色 / 余白 / 角丸は `styles/base.css` の CSS 変数（トークン）経由で指定し、値の直書きをしない。盤の見た目は `board.css`、それ以外は `base.css`。

## ドキュメンテーション（JSDoc）
- **DOM に触れない層（`rules.js` / `game.js` / `cpu.js`）の公開関数には JSDoc を書く。** 仕様変更の起点になるため、引数・戻り値・副作用を型付きで明示する。
- 最低限 `@param`（型と意味）・`@returns`（型と意味）。状態を書き換える関数はその旨を1行添える。
- 状態の形（`State` など）は `@typedef` で1回定義し、各関数から参照して重複を避ける。
- `view.js` / `main.js` は必須にしない（意図が非自明な箇所のみコメントで補う）。
- 新規の公開関数は必須。既存の未記述関数は、そのファイルを触ったついでに追記する（一括対応はしない）。

## 安全・堅牢性
- 文字列を DOM に出すときは `textContent` を使う。`innerHTML` は**このリポジトリが書いた固定文字列のみ**に限る（ログ・案内文に外部入力を混ぜない）。将来入力を受け付けるなら必ずエスケープする。
- 演出中の入力は `state().busy` でロックする。二重着手・演出の重なりを許さない。

## スタイル
- 既存コードの記法・命名・コメント密度に合わせる。新しい流儀を持ち込まない。コメントは日本語。

## オーバーエンジニアリングの防止

方針: 「**まず動くものを作り、気になったら作り直す**」。未決事項は最もシンプルな選択で確定する。

- **YAGNI**: いま必要な機能だけ作る。[`spec.md`](spec.md) §5 で「やらない」「未決」に置いた項目を先取りしない。
- **抽象化は後出し**: 設定オプション・汎用エンジン・プラグイン機構を「将来のため」に作らない。共通化は2か所以上で必要になってから。
- **最小の選択**: 仕様が曖昧な点は最も単純な実装を選び、判断を一言メモして進む。
- **巨大化対策**: ファイルが肥大したら責務で分割するが、分割のための分割はしない。

迷ったら「これは本当に要るか？ spec のどこに根拠があるか？」を先に確認する。
