# CLAUDE.md

このリポジトリで作業する際の**運用ガイド**。CLAUDE.md は日々の作業で常に守る手順・必須ルール（開発ワークフロー）を扱い、**ゲームルール・構成などの詳細仕様は [`spec.md`](spec.md) を単一ソース（正）とする**。二重記載を避けるため、詳細は本書に書き写さず各ドキュメントへリンクする。記述が食い違ったら [`spec.md`](spec.md) を優先する。

## プロジェクト概要

**トラップ・オセロ** — 通常のオセロに「伏せた罠」と「見切り」を足した、対 CPU の1人用ゲーム（プロトタイプ）。

- エントリ: [`index.html`](index.html)（これをブラウザで開く）＋ `styles/*`（見た目）＋ `src/*`（ロジック）
- 仕様: [`spec.md`](spec.md) — ゲームルール・CPU の方針・構成とレイヤーの契約（**本書より spec.md を優先**）
- コーディング規約: [`CODING.md`](CODING.md) — 言語/実行環境・モダン JS・構造・JSDoc・オーバーエンジニアリング防止（**コードを書く前に参照**）
- データ永続化・外部送信: **行わない**（1試合ごとにメモリ上で完結）

設計思想（重要）: **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない。単一 HTML ではなく **「エントリ HTML ＋ スタイル ＋ 責務別スクリプト」** 構成とする（[`spec.md`](spec.md) §4 / [`CODING.md`](CODING.md)）。

## 構成（作業で常に守ること）

詳細は [`spec.md`](spec.md) §4.1 の表を正とする。ここでは常に守る運用ルールだけを挙げる:

- **依存は一方向**（`config → rules → game → cpu → view → main`）。逆向きの依存を作らない。
- **状態は `game.js` に一元化する**。`view` / `main` / `cpu` が独自に対局状態を持たない。
- **`rules` / `game` / `cpu` は DOM に触れない**。DOM を使いたくなったら、その処理は `view` か `main` の責務。
- **調整値は `config.js`**。枚数・手数・待ち時間・評価の重みをロジックへ直書きしない。
- 機能を足すときは、まず「どのレイヤーの仕事か」を決めてからファイルを開く。置き場所に迷う処理は、たいてい責務がずれている。

## 動作確認

ビルド不要。[`index.html`](index.html) をブラウザで直接開く（`file://`）か、任意の静的サーバーで配信する。

```
npx --yes http-server -p 8777 -c-1
```

自動テストは用意していない。**ロジック（`rules` / `game` / `cpu`）を変更したら、次を必ず確認する:**

1. コンソールにエラーが出ないこと。
2. 1局を通して遊べること（罠の設置 → 発動 → 横取り、見切りの成功と空振り、パス、終局まで）。
3. ロジックを大きく触ったときは、DevTools のコンソールで自動対局を回して例外が出ないことを確認する:

   ```js
   // 双方ランダム／CPU で20局回す（演出を挟まないので一瞬で終わる）
   const { game, cpu, rules, config } = window.TO;
   for (let t = 0; t < 20; t++) {
     game.newGame(); const S = game.state(); let guard = 0;
     while (S.phase === "play" && guard++ < 200) {
       const me = S.turn;
       if (me === config.CPU) cpu.arm();
       const mv = me === config.CPU ? cpu.chooseMove() : null;
       const list = [...rules.legalMoves(S.bd, me).keys()];
       if (!list.length) { game.endTurn(); continue; }
       const idx = mv ? mv.idx : list[(Math.random() * list.length) | 0];
       const info = game.applyMove(idx, me, mv ? mv.shield : false);
       if (info.fired) game.doSteal(info);
       game.endTurn();
     }
     console.log(S.phase, rules.counts(S.bd).join("-"), "ply" + S.ply);
   }
   ```

## 開発ワークフロー（必須ルール）

以下は**必ず守る**こと:

1. **作業は機能単位で Issue を立ててから始める。** 実装の前に GitHub Issue を作成し、何を作るか・受け入れ条件を明記する。書式は下記「Issue の書き方」に従う。
2. **作業はブランチを切って行い、PR を出す。** 直接 `main` にコミットしない。PR 本文に `Closes #<issue番号>` を入れて Issue と紐付ける。
3. **PR を出した後、必ず `/code-review` を実施する**（ユーザーがレビュー省略を指示したときはその限りでない）。指摘はマージ前に対応するか、対応しない場合は PR コメント or 別 Issue として残す。
4. **`/code-review` の指摘は次のサイクルで処理する。** 1巡は (a)〜(c) の3ステップ。
   - (a) 指摘内容を PR コメントとして残す（対応する／しないの方針も記載）。
   - (b) 対応すると判断した指摘を修正してコミット・プッシュする。
   - (c) 再度 `/code-review` を実施する。

   **ループの終了条件（必ず守る）:**
   - **自律的に回してよいのは3巡まで。その3巡は毎巡止まらず続けてよい。** 3巡を終えても **High / Medium** が残るなら人間の確認が必要と判断し、そこで止まって結果を要約して報告し、指示を仰ぐ。
   - **「3巡」は1回の許可あたりの予算であり、通算の上限ではない。**「実装に入って」のような実装の指示が最初の3巡ぶんの許可にあたる。再開の指示が出たら、そこから新たに3巡ぶん回してよい。
   - **予算になるのは「レビューの継続を含む指示」だけ**（「続けて」「もう一度回して」等）。「この指摘だけ直して」のような個別の修正指示は予算にならない（直して報告し、そこで止まる）。
   - **High / Medium がゼロになったらそこが終点。** 3巡を使い切っていなくても打ち切る。Low を潰すために追加で回さない。
5. **参照したドキュメントが実装・現状と食い違っていたら放置しない。** 軽微なら**同じブランチ／PR で更新する**。スコープを超える場合は `documentation` ラベルの Issue を立てて残す。ドキュメントの鮮度は「触った人がそのとき直す」ことで維持する。
   - **spec の未決事項を確定させた PR は、該当記述を「実装済み」注記に書き換えるのではなく現在形へ刈る**（済んだ記述を積まない。履歴は git に任せる）。
   - **定数・ファイル・関数を改名したら、その名前でドキュメント全体（`spec.md` / `README.md` / `CODING.md` / `CLAUDE.md`）を grep して同期する。** 「読んだ箇所だけ直す」では取り逃す。

### Issue の書き方

**Issue を立てるセッションと実装するセッションは別で、壁打ちの内容は実装側に引き継がれない**前提で、Issue 単独で読んで実装できるように書く。

**タイトル・ラベル**

- **タイトルは領域接頭辞で始める**: `[<領域>] <動詞から始まる要約>`。
  - 使う接頭辞: `[rules]`（ゲームルール・罠/見切りの仕様）｜`[cpu]`（思考）｜`[ui]`（描画・演出・レイアウト）｜`[core]`（構成・仕様・ドキュメント等、特定領域に属さないもの）。
  - 例: `[rules] 見切りの空振り時に手札を消費しない不具合を直す` / `[cpu] 終盤の読みを2手先まで広げる` / `[ui] 罠発動バナーを狭幅で収める`。
- **種別は GitHub ラベルで示す**: バグ＝`bug`、機能追加・改善＝`enhancement`、ドキュメントのみ＝`documentation`。新しいラベルをむやみに増やさない。

**本文の大項目（種別共通・この順で固定）**

| 大項目 | 中身 | 必須/任意 |
|---|---|---|
| `## 概要` | 何を・なぜ。1〜3行の要約 | 必須 |
| `## 背景` | **なぜ**（経緯・現状の課題）。単独で読んで文脈を掴める厚みで書く | 必須 |
| `## 詳細` | **何を**（やること）。バグは `### 再現手順`・`### 期待`・`### 実際` を立てる | 必須 |
| `## 決定事項` | 壁打ちで確定した方針・判断（採否理由込み）。無ければ「特になし」と書く | 必須（常設） |
| `## スコープ外` | 今回やらないこと | 任意 |
| `## 受け入れ条件` | 完了／修正の判定基準（チェックボックス `- [ ]`） | 必須 |
| `## 参考` | 関連 spec / Issue / PR | 任意（推奨） |

- **背景と詳細を混ぜない**: 背景＝「なぜ」、詳細＝「何を」。
- **決定事項は常設**: 実装セッションは壁打ちを知らないので、確定事項はここに残す。無ければ「特になし」と明記して未決を可視化する。

### PR の書き方

| 大項目 | 中身 | 必須/任意 |
|---|---|---|
| `## 概要` | やったこと・なぜ | 必須 |
| `## 変更` | 具体的な変更点 | 必須 |
| `## テスト` | 実施内容と結果（上記「動作確認」の何をどこまで確認したか） | 必須 |
| `## 受け入れ条件` | Issue のチェックリストを転記し、対応状況を反映（`- [x]`） | 必須 |
| `Closes #<issue番号>` | Issue 紐付け | 必須 |

- 見出しは Issue も PR も `## 概要` / `## 変更` に統一する。PR に `## 背景` は付けない（Issue 側で足りる）。

### 大項目を埋めるときのルール（推測で埋めない／勝手に決めない）

- **必須の大項目は必ず立てる。埋められない場合は省略・推測せず、ユーザーに確認する。** 特に `## 背景` が薄いまま Issue を立てない。
- **実装時に Issue の `決定事項` に無い設計判断が必要になったら、勝手に決めず着手前にユーザーへ確認する。**

### 補足

- **止まる境目は「元に戻しやすいか」で引く。不可逆なのはマージだけなので、明示的な指示を待つのもマージだけ。** ブランチ作成・コミット・プッシュ・PR 作成は実装の指示に含まれるものとして進めてよい。
- コミットメッセージ・PR・Issue は日本語で記述する。
- コードを書くときの規約・禁止事項は [`CODING.md`](CODING.md) を参照する（**ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない）。
