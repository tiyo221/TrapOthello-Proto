# spec.md — トラップ・オセロ 仕様

**何を作るか**の単一ソース（正）。実装・他のドキュメントと食い違ったら本書を優先する。**どう作るか**は [`CLAUDE.md`](CLAUDE.md)（運用）と [`CODING.md`](CODING.md)（コーディング規約）を参照。

## 1. 概要

通常のオセロに**「伏せた罠」**と**「見切り」**を足した1人用（対 CPU）のゲーム。相手の着手そのものを利益に変える読み合いを主眼とする。

- プレイヤー＝黒（先手）、CPU＝白。
- 盤・裏返しのルールは標準のオセロに従う（8×8、初期配置は中央4石）。
- 勝敗は終局時の石数で決める。

### 1.1 設計思想

**ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない。プロトタイプとして「まず遊べること」を優先し、凝った機構を先回りして作らない（[`CODING.md`](CODING.md) のオーバーエンジニアリング防止）。

## 2. ゲームルール

### 2.1 罠

| 項目 | 値 | 定数 |
|---|---|---|
| 手札（1試合の総数） | 3個 | `TRAPS_HAND` |
| 1手番に伏せられる数 | 1個 | `ARM_PER_TURN` |
| 設置可能になる手数 | 制限なし（0手目から） | `ARM_FROM_PLY` |

- 自分の手番に、**石を置く前**に空きマスへ1個まで伏せられる。
- 罠は**プレイヤーごとに独立したレイヤー**に置かれる。同じマスを両者が伏せても互いに残り、**相手の罠だけ**が自分に反応する。
- 自分の罠を自分で踏んだ場合は、発動せず**無効化されるだけ**（手札は戻らない）。
- 自分の罠は盤上に見える（金の菱形）。相手の罠は**位置が見えない**（残数だけスコア欄に「不明」付きで公開）。

### 2.2 罠の発動（2段階解決）

罠のあるマスへ相手が着手したとき、**その手をまず通常どおり解決してから**成果を丸ごと設置者が奪う。演出上もこの2段階を見せる。

1. **第1段階**：着手を通常のオセロとして解決する（置いた石＋裏返った石が着手側の色になる）。
2. **第2段階**：置いた石と裏返った石を**すべて設置者の色に変える**。罠は消費され、そのマスは発動済みとして公開される。

石数の差は `枚数 × 2` 動く（`枚数` ＝ 置いた石1枚＋裏返った石）。

### 2.3 見切り

| 項目 | 値 | 定数 |
|---|---|---|
| 使用回数 | 1試合1回 | `SHIELD_COUNT` |

- 宣言したターンのみ、相手の罠を無効化する。
- 踏んだ罠は**解除・公開**され、通常解決の成果はそのまま着手側が確保する。
- **空振り（罠が無かった）でも消費**する。
- **使用は相手に見える**（残数は双方公開）。

### 2.4 進行・終局

- 打つ手が無い側はパスする。双方に打つ手が無くなったら終局し、石数の多い側の勝ち（同数なら引き分け）。

## 3. CPU

思考は難易度ごとに `src/cpu/<id>.js` へ分ける（**初級**（`beginner`）／**中級**（`intermediate`）／**上級**（`advanced`）の3つ）。合法手の列挙・盤面評価（マス評価と着手可能数の差）・着手の選択・見切りの適用は難易度に依らない共通処理として `cpu.js` が持ち、難易度は「罠を伏せるマス・着手の加点・見切りの可否」だけを実装する。3つとも探索木は持たず一手先の評価のみで、盤の基礎評価は共通（難易度差は探索の深さではなく**罠と見切りの使い方**にある）。既定の難易度は `CPU_LEVEL_DEFAULT`（＝中級）。

いずれも `TO.game.publicView(color)` が返す公開情報だけで判断し、**相手の罠の位置は参照しない**（相手の罠は数だけ見える）。共通の評価は次のとおり。

- **中盤**：マス評価値（`WEIGHTS`）＋着手可能数の差（`CPU_MOBILITY_WEIGHT * (自分 - 相手)`）。
- **終盤**（空きマスが `CPU_ENDGAME_EMPTIES` 以下）：石数差を最優先。
- 同点手の揺らぎとして小さな乱数（`CPU_TIE_NOISE`）を加え、毎回同じ対局にならないようにする。

**初級**（`BEGINNER.*`）— 公開情報を使い切らない素朴な相手。相手が罠を伏せていても、**踏むリスクを着手の評価に織り込まない**（相手の伏せ数を見るのは見切りの判断だけ）。

- **罠への誘導**：着手後に相手が自分の罠へ踏み込める場合、期待枚数で加点する（`LURE_*`）。相手の合法手が**すべて自分の罠**になる局面は大きく加点する（`ALL_TRAPPED`）。
- **罠の設置**：既存の石に隣接するマス（近いうちに戦場になる）だけを候補にし、スコアが `ARM_MIN` 以上なら伏せる。`ARM_FORCE_PLY` 手を過ぎたら残りを使い切る。
- **見切り**：相手が罠を伏せている状態で、角を取りに行く／`SHIELD_MIN_FLIPS` 枚以上裏返す／`SHIELD_FORCE_PLY` 手目以降のいずれかで宣言する。

**中級**（`INTERMEDIATE.*`）— 初級の誘導・設置を引き継ぎつつ、**公開情報を使い切る**。踏まされる回数・奪われる枚数で初級を明確に下回る（対人間プロキシの計測で確認）。

- **踏むリスクの見積り**：相手の伏せ数が 0 でないとき、着手の評価から期待損失を引く（`伏せ数 ÷ 空きマス数 × 裏返り枚数 × 2` を石差とみなす粗いモデル、`RISK_DISC_WEIGHT` で換算）。大きく裏返る手ほど不利に評価される。
- **見切りの判断を期待損失ベースにする**：上の期待損失が `SHIELD_MIN_RISK` を超えたら切る（`SHIELD_FORCE_PLY` 手目以降は使い残さない）。
- **相手の見切り残数を使う**：相手の見切りが尽きていれば罠は確実に効くので、設置の下限を `ARM_MIN` から `ARM_MIN_NO_SHIELD` へ下げて強気に伏せる。残っている間は、相手が見切りを切りやすい角へ伏せる価値を `ARM_CORNER_VS_SHIELD` だけ下げる。

**上級**（`ADVANCED.*`）— 中級の判断（誘導・リスク見積り・見切り）を**すべて引き継いだ上で、罠と見切りの運用だけを変える**。盤評価・探索の深さは中級と共通。

- **温存**：`ADVANCED.ARM_FROM_PLY`（＝50手目前後）まで罠を伏せない。早すぎると中盤の石数（＝横取りの通貨）が弱く、遅すぎると死に札が増えるため、その手前に最適がある。
- **被せ方**：自分の着手を決めたあと、その着手後の盤で**相手の合法手**を計算し、最も裏返りが大きくなるマスを覆う（踏めば大きく横取りできる）。覆える相手の合法手が無ければ、マス評価が高い空きへ逃がして死に札を防ぐ。
- **見切りの温存**：`ADVANCED.SHIELD_FROM_PLY` までは見切りを切らず、終盤に相手が合法手を覆ってきたときの突破口として残す。それ以降は中級と同じ期待損失で判断する。

### 3.1 難易度の選択

- サイドパネルの難易度セレクトで 初級／中級／上級 を選ぶ。既定は**中級**（`CPU_LEVEL_DEFAULT`）。
- 対局中に変更しても**進行中の対局は変わらず**、次の「最初から」で反映する（進行中の対局と CPU の方針が食い違うのを防ぐ）。選択が現在の対局と食い違っている間は、その旨をセレクト下に注記する。

## 4. 構成

単一ファイルではなく **「エントリ HTML ＋ スタイル ＋ スクリプト（責務別）」** 構成とする。`file://` で確実に動かすため **ES Modules / `fetch()` は使わず**、classic `<script src>` ＋ グローバル名前空間 `window.TO` で連携する。

```
index.html      … マークアップと読み込み順の宣言のみ（ロジックを書かない）
styles/
  base.css      … デザイントークン・レイアウト土台・サイドパネル
  board.css     … 盤・石・罠・演出
src/
  config.js     … 盤の定数と調整用パラメータ（TO.config）
  rules.js      … オセロの純粋ルール（TO.rules）
  game.js       … 対局状態と状態遷移（TO.game）
  cpu.js        … CPU のファサードと共通処理（TO.cpu）
  cpu/
    beginner.js     … 初級の判断（TO.cpuLevels.beginner）
    intermediate.js … 中級の判断（TO.cpuLevels.intermediate）
    advanced.js     … 上級の判断（TO.cpuLevels.advanced。中級へ委譲し罠と見切りの運用だけ変える）
  view.js       … 描画・アニメーション（TO.view）
  main.js       … 進行制御・入力・起動（公開シンボルなし）
```

### 4.1 レイヤーの契約

読み込み順は上から下（下ほど上に依存する）。**逆向きの依存を作らない。**

| ファイル | 依存 | DOM | 責務 |
|---|---|---|---|
| `config.js` | なし | × | 定数・調整値の単一ソース。バランス調整はここだけを触る |
| `rules.js` | config | × | 引数の盤面だけを見る純粋関数（状態を持たない） |
| `game.js` | config, rules | × | 対局状態 `State` の保持と状態遷移（罠・見切り・パス・終局） |
| `cpu.js` | config, rules, game | × | 思考のファサードと共通処理（合法手の列挙・盤面評価）。罠の設置だけ副作用を持つ |
| `cpu/<id>.js` | config, rules, cpu | × | 難易度ごとの判断（罠を伏せるマス・着手の加点・見切りの可否）。`game` に触れない。**上級のみ例外的に中級（同じ葉）へ委譲する**（罠と見切りの運用だけを差し替え、盤評価は中級と共有するため） |
| `view.js` | config, rules, game | ○ | 状態を読んで描くだけ。対局ロジックを持たない |
| `main.js` | 全部 | ○ | 入力の受け取り・演出の間合い（`setTimeout`）・手番の受け渡し |

- **状態は `game.js` が一元的に持つ**（`TO.game.state()`）。他のレイヤーは自前で対局状態を複製しない。
- **CPU は非公開情報を見ない。** 難易度の実装が受け取るのは `TO.game.publicView(color)` が返す公開情報だけで、**相手の罠の位置は含まれない**（相手の罠は「数」まで）。CPU 系のうち `game` に触れるのは `cpu.js` だけで、難易度側は `game` を参照しない。**これは規約であり、`file://` で動く classic script である以上グローバル（`window.TO`）は難易度からも見える。** 構造で不可能にはできないので、[`CLAUDE.md`](CLAUDE.md) の動作確認で `grep` して担保する。
- **`view.js` は判定をしない**（勝敗・合法手の判断は `rules` / `game` に委ねる）。
- **`game.js` / `cpu.js` は `document` / `window` の DOM API に触れない**（そのまま Node で動かせる純粋さを保つ）。
- **演出のタイミング（`setTimeout`）は `main.js` に集約する**。待ち時間の値は `config.js`（`DELAY_*`）。

## 5. 未決事項・やらないこと

- **やらない**：オンライン対戦・データの永続化・外部送信。
- **未決**：探索深さを増やした CPU、罠の種類追加。必要になってから spec を更新して着手する（先取りしない）。
